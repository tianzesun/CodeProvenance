# AI Development Rules for CodeProvenance

## Core Principles

1. **Type Safety First**: Use Python type hints for all function parameters and return values
2. **Single Responsibility**: Each module/class/function should have one clear purpose
3. **No Magic Numbers**: Use named constants instead of hardcoded values
4. **Error Handling**: Always handle exceptions gracefully with appropriate error messages
5. **No Debug Code in Production**: Remove print statements, use proper logging

---

## Code Style Guidelines

### Python Standards
- Follow PEP 8 style guide
- Maximum line length: 100 characters
- Use 4 spaces for indentation (no tabs)
- Use `from __future__ import annotations` for forward references
- Use dataclasses or Pydantic models for data structures

### Naming Conventions
| Type | Convention | Example |
|------|------------|---------|
| Variables | snake_case | `submission_id` |
| Functions | snake_case | `calculate_similarity()` |
| Classes | PascalCase | `SimilarityResult` |
| Constants | UPPER_SNAKE | `MAX_FILE_SIZE` |
| Private | _prefix | `_internal_cache` |

### Docstrings
```python
def calculate_similarity(file1: str, file2: str) -> float:
    """
    Calculate similarity score between two code files.
    
    Args:
        file1: Path to the first file
        file2: Path to the second file
    
    Returns:
        Similarity score between 0.0 and 1.0
    
    Raises:
        FileNotFoundError: If either file does not exist
        ValueError: If files are empty
    """
```

## Project Structure Rules

### Module Organization
- All source code in `src/` directory
- Import order: stdlib → third-party → local
- Use relative imports within the package
- No circular imports

### Configuration
- All configurable values in `config/` module
- Environment variables for secrets
- Use Pydantic for config validation
- No hardcoded paths or URLs

### API Design
- RESTful endpoint design
- Request validation with Pydantic schemas
- Consistent error response format
- Rate limiting for all endpoints

---

## Testing Requirements

### Test Coverage
- Minimum 80% code coverage for core modules
- 100% coverage for similarity algorithms
- All API endpoints must have integration tests

### Test Structure
```
tests/
├── unit/
│   ├── test_parser.py
│   ├── test_processor.py
│   └── test_similarity.py
├── integration/
│   └── test_api.py
└── fixtures/
    └── sample_submissions/
```

### Test Naming
- Test files: `test_<module_name>.py`
- Test functions: `test_<description>_<expected_behavior>`
- Use descriptive assert messages

---

## Security Rules

1. **Input Validation**: Validate ALL input before processing
2. **Path Traversal**: Prevent directory traversal attacks in file operations
3. **File Size Limits**: Enforce maximum file upload sizes
4. **Resource Limits**: Set timeout for all operations
5. **No Secrets in Code**: Use environment variables for all secrets
6. **Sanitize Output**: Escape all output before returning

---

## Performance Rules

1. **Async First**: Use async/await for I/O operations
2. **Connection Pooling**: Reuse database/network connections
3. **Caching**: Cache expensive computations with TTL
4. **Streaming**: Use generators for large file processing
5. **Batch Processing**: Process submissions in batches when possible

## Git Workflow

### Commit Messages
```
<type>: <short description>

Types: feat, fix, refactor, test, docs, chore

Example:
feat: add file parsing for Java submissions
fix: handle empty files in similarity calculation
docs: update API documentation
```

### Branch Naming
- `feature/<description>` - New features
- `fix/<description>` - Bug fixes
- `refactor/<description>` - Code improvements
