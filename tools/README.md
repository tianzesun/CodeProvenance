# Tools Directory

This directory stores third-party tools, helper libraries, tool configs, and runtime artifacts used by the benchmark framework.

## Purpose

The benchmark compares multiple external code-similarity and plagiarism-detection systems under a common protocol. To keep the benchmark reproducible and maintainable, raw tool assets are separated from Python adapter code and benchmark orchestration. Detailed metadata about tools, versions, paths, and execution settings should be recorded for every benchmark run.

## Layout

```text
tools/
├── external/   # standalone third-party tools or repos
├── libs/       # helper libraries used inside internal baselines
├── configs/    # per-tool config templates and defaults
├── sandbox/    # temporary execution workspaces
├── outputs/    # raw tool outputs
└── README.md
```

## Folder meanings

### external/
Contains independently runnable tools that may appear as benchmark entrants.

Examples:
- moss
- JPlag
- NiCad-6.2
- dolos
- pmd
- Sherlock
- vendetect-0.0.3

These tools should only be used through registered adapters, not called ad hoc from benchmark logic. Adapter-based benchmarking works best when external tool invocation is standardized.

### libs/
Contains reusable libraries used to implement lightweight internal baselines.

Examples:
- textdistance
- fuzzywuzzy
- py_stringmatching
- sim_metrics

These are not automatically benchmark entrants.

### configs/
Contains per-tool configuration files, ignore lists, starter-code exclusions, and language-specific templates. Recording configuration inputs is important for reproducibility.

### sandbox/
Temporary per-run execution directories. Tools that need copied datasets, generated input layouts, or isolated workspaces should run here.

### outputs/
Raw outputs generated by external tools, such as HTML, XML, JSON, logs, or text reports. Raw outputs should be preserved separately from normalized benchmark results so later analysis remains traceable.

## Official benchmark participants

The benchmark should expose a curated set of official tools through the Python tool registry. Presence inside `tools/external/` does not automatically make a tool an official benchmark participant. Benchmark scope should be explicit and documented.

## Adding a new tool

1. Place the tool under `tools/external/` if it is independently runnable, or under `tools/libs/` if it is only a helper library.
2. Add a matching adapter in `src/backend/benchmark/adapters/`.
3. Register the tool in the benchmark tool registry.
4. Record tool version, path, supported languages, and config defaults.
5. Verify the tool can run reproducibly on benchmark datasets.

## Reproducibility notes

Every benchmark run should record:
- tool name
- tool version
- external path
- adapter name
- enabled/disabled state
- dataset version
- config file or arguments
- execution timestamp
- benchmark commit hash if available

Reproducibility depends on preserving metadata about tools, versions, dependencies, and execution settings.
